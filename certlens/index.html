How CertLens works

Every public SSL/TLS certificate is logged in Certificate Transparency — a public, append-only system designed to detect mis-issuance. Anyone can search this data.

CertLens uses the SSLMate CertSpotter API to retrieve certificates for a domain (and all of its subdomains), then runs DNS lookups via Google's public DNS-over-HTTPS resolver to see which hosts are actually responding.

Scan history

Your last 10 completed scans are stored in your browser (localStorage only — nothing sent anywhere). Click any chip in the history strip to instantly revisit a past scan without re-fetching from the CertSpotter API. Most-recent scans are shown first.

Executive summary & PDF export

The Executive Summary panel surfaces a plain-English overview and the top 5 prioritized actions tied to specific hosts. Use Export → Print summary (PDF) for a 2-3 page report suitable for leadership; use Print full report (PDF) to include all hosts, findings, and the issuance timeline. Use Copy as Markdown for pasting into Jira, Slack, Confluence, or GitHub issues.

Watchlist & change detection

Save domains to your local watchlist to track changes over time. CertLens stores known certificate IDs in your browser and flags newly-discovered certificates on each rescan. Export and import your watchlist as JSON to move it between machines.

What CertLens flags

Suspicious naming patterns (login-, secure-, verify-), abnormal issuance frequency for a single host, revoked certificates, expiring or expired certs, and mixed-issuer patterns that may indicate inconsistent ownership.

Honest limits

Wildcard certificates (*.example.com) don't reveal subdomain names. The unauthenticated CertSpotter API has hourly rate limits — heavy use of "Maximum" depth on many domains may hit them. Certificates from private PKI never appear in public CT logs.
