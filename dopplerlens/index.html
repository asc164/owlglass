How DopplerLens works

DopplerLens uses the same family of algorithms as dnstwist — character omission, duplication, transposition, keyboard-adjacent substitution, homoglyph substitution, character insertion, hyphenation, TLD swaps, common prefixes/suffixes, and bitsquatting (single bit-flip mutations). Each algorithm produces variations of your domain that humans might mistype or fail to notice.

For each candidate, DopplerLens runs DNS lookups via Google's public DNS-over-HTTPS resolver to see if the domain is registered. Live candidates get deeper analysis: nameservers, mail servers, web server reachability, and a favicon hash comparison against your domain.

How verdicts are assigned

Each domain accumulates a suspicion score from positive and negative signals. Sharing your IP, nameservers, or mail servers reduces the score (defensive). Having an active web server on different infrastructure raises it. Matching favicon on different infrastructure raises it sharply. Brand-protection registrars (MarkMonitor, CSC) are recognized and scored as defensive even without infrastructure overlap.

Honest limits

DopplerLens does not detect IDN homograph attacks using non-Latin Unicode (e.g. Cyrillic letters that look identical to Latin ones) — those require punycode-aware generation that is unreliable from the browser. Favicon comparison only works when the suspect domain serves its favicon with permissive CORS headers, which most don't, so absence of a favicon match is not exonerating evidence. The DNS-based liveness check can produce false positives for shared parking infrastructure where many unrelated dormant domains resolve to the same IP.
